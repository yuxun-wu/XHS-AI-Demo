# 小红书收藏夹智能分类 · 产品 Demo

> 一个用 HTML 单文件模拟的小红书收藏分类工具 demo，回答的不是"怎么分类"，而是"AI 主动做事时，边界该划在哪里"。

迭代 7 版，全部交互流程跑通，可直接打开浏览器演示。

---

## 一、它在做什么

模拟小红书「我的收藏」场景，让用户能用两种方式整理自己的收藏笔记：

- **我来定专辑**：自己输入想要的分类名，AI 帮你归位
- **AI 自动归类**：AI 通读笔记，自动生成专辑体系

但 demo 真正的重点不在分类算法本身，而在于把一次完整的分类操作拆成五个状态：

```
选模式 → 选策略（已有分类时）→ 选粒度（AI 模式时）→ 草稿态 → 应用态
```

每一步都在回答同一个问题：**这一步该让 AI 替用户决定，还是该停下来问一句？**

---

## 二、怎么打开

直接用浏览器打开 `xhs-demo-v7.html` 即可，无需任何依赖、无需启动服务。

推荐浏览器：Chrome / Edge / Safari 最新版。

## 三、四个模拟用户

页面顶部切换条提供 4 个用户，代表 4 种真实存在的初始状态：

| 用户 | 收藏数 | 专辑数 | 已分类 | 用来演示什么 |
|------|--------|--------|--------|--------------|
| 小安 | 60 | 0 | 0 | 全新用户首次分类 |
| 小贝 | 62 | 2 | 0 | 建了空专辑、还没动手分类 |
| 小晨 | 65 | 4 | 22 | 分类分到一半、剩下一堆未归类 |
| 小丁 | 68 | 11 | 68 | 已完整分类过，演示"再来一次" |

切换用户时整个工作流和瀑布流都会同步刷新。用户标签（`已分类 22/65 篇 · 4 个专辑`）会随分类操作实时更新。

## 四、核心交互流程

### 1. 入口

手机端「我的收藏」页面顶部永远有一个粉色 banner「✨ 智能分类」，点击滚到右侧控制面板。Tab 只有「全部」「专辑」两栏，可点击切换。

### 2. 前置询问（仅对已有分类的用户）

小贝 / 小晨 / 小丁 进入分类流程时，先问一步：
- **在已有基础上整理**：原有专辑作为锁定 chip，AI 只处理未归类的笔记
- **从头重新分类**：忽略原有结果，按全新方式重做

新用户（小安）跳过这步。

### 3. 选分类方式

- **我来定专辑**：手动输入类目名，支持快速添加预设
- **AI 自动归类**：进入下一步选粒度

### 4. AI 粒度选择（仅 AI 模式）

横排三选项：粗略 / 平衡 / 细分。每个选项展示：
- 「≈ N 个专辑」徽章（根据当前用户笔记动态预估）
- 两个用户**自己收藏里真实会被分出来**的类目作为示例

例如小安选粒度时看到的是「美食 / 旅行 / 家居」，小贝看到的可能是「化妆美妆 / 护肤抗痘 / 日常穿搭」——示例跟着用户的收藏走，每个人看到的都不一样。

### 5. 草稿态

点「生成草稿」后，**不会立刻修改用户收藏夹**。结果以草稿形式呈现：
- 顶部黄色 sticky 确认条，展示 diff（`+5 新归类 ~3 变更 -1 移除 8 未归类`）
- 瀑布流的角标变成黄色虚线半透明，区别于已生效的实色黑底
- 入口 banner 提示"分类草稿已生成"

草稿态下可以：
- **重命名**专辑（双击或点 ✎）
- **合并**专辑（点 ⇄，会弹出推荐新名字的对话框）
- **删除**专辑（点 🗑）
- **采纳建议**：手动模式下 AI 会建议补充类目，可以一键采纳重做

### 6. 应用确认

点「✓ 应用到我的收藏夹」时：

- **如果有未归类笔记**：弹出模态框问要不要建「其他」专辑统一存放
  - 「建『其他』专辑」：所有未归类笔记进入新建的兜底专辑
  - 「保持未归类」：这些笔记保持无专辑状态
- **如果全部归类完毕**：直接应用

应用后才真正写入 `u.applied`，瀑布流角标变实色，标签实时更新。

### 7. 合并的智能命名

合并时不再让一个专辑直接吞掉另一个，而是弹出对话框：

```
[力量训练 5篇]  +  [有氧训练 3篇]

✨ 推荐名称   [基于上位概念]
┌────────────────┐
│ 健身             │  ← 可编辑
└────────────────┘

[✨ 健身]  [有氧训练]  [力量训练]
```

推荐算法分四层：
1. **语义层级表**（60+ 条子→父映射）：两专辑共享父节点时推荐父节点
2. **父子关系**：一方是另一方的父
3. **共同后缀**：「力量训练 + 燃脂训练」→「训练」
4. **共同前缀**：「日本旅行 + 日本美食」→「日本」

用户始终掌握最终编辑权——推荐只是默认值。

## 五、推荐的体验路径

要在 5 分钟内看完所有核心特性，按这个顺序走：

1. **小安**（全新用户）→ AI 自动归类 → 试一下三个粒度看示例和数量怎么变 → 选细分 → 生成草稿 → 在草稿里合并两个专辑（比如「力量训练」+「燃脂HIIT」→ 看到推荐「训练」）→ 应用
2. **小晨**（部分分类）→ 看到前置询问 → 选「在已有基础上」+ 手动模式 → 看到原 4 个专辑是蓝色锁定 chip → 自己加一两个新专辑 → 生成草稿
3. **小贝**（空专辑）→ 我来定专辑 → 只输入「美食」一个 → 生成草稿 → 应用时会看到「N 篇未归类」弹窗 → 选「保持未归类」看效果
4. 切「专辑」Tab 看每个用户的专辑卡片视图（4 宫格封面拼图）

## 六、代码结构

单文件 HTML，约 3500 行。结构如下：

```
<head>
  └── <style>           样式（粒度卡、专辑卡、modal、合并对话框等）

<body>
  ├── 用户切换条
  ├── 手机模拟器（左侧）
  │     ├── 顶部 Tab（全部 / 专辑）
  │     ├── 智能分类入口 banner
  │     └── 瀑布流 / 专辑列表 / 专辑详情（按 Tab 切换）
  └── 控制面板（右侧，所有分类工作流的核心交互区）

<script>
  ├── allNotes[]            80 条模拟笔记池
  ├── users[]               4 个模拟用户（含 collectedIds + applied 状态）
  ├── synonymMap            手动模式的关键词扩展词典
  ├── granularityPresets    三套粒度的 categoryMap（粗略 / 平衡 / 细分）
  ├── categoryHierarchy     60+ 条子→父语义层级表（合并推荐用）
  │
  ├── runClassify()         核心分类算法（按 tag 命中得分）
  ├── previewGranularity()  根据用户笔记预估粒度生成几个专辑
  ├── suggestMergedName()   合并时的智能命名推荐
  ├── computeDiff()         草稿 vs 已应用的 diff 计算
  │
  ├── renderUserSwitcher()  用户切换条
  ├── renderPhoneBanner()   手机端入口 banner
  ├── renderWaterfall()     瀑布流（根据 phoneTab 派发到三种视图）
  ├── renderPanel()         右侧面板（按 state 决定渲染 setup/processing/results）
  ├── renderGranularityPicker()  粒度选择卡片
  ├── renderPendingResults()     草稿态结果展示
  ├── showUnclassifiedDialog()   未归类二次询问弹窗
  ├── showMergeRenameDialog()    合并推荐名对话框
  │
  └── state{}               全局状态（当前用户、phoneTab、mode、granularity、
                                    strategy、pending、suggestions 等）
```

所有状态变更后调用 `renderAll()` 统一重渲染，没有局部刷新逻辑——简单但够用。
